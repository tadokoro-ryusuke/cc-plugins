## なぜdescriptionが重要か

descriptionはスキルトリガーの**唯一の判断材料**。Claudeは以下の順序でスキルを処理:

1. 全スキルのname + descriptionをシステムプロンプトに読み込み
2. ユーザークエリに基づき、descriptionからスキルを選択
3. 選択されたスキルのSKILL.md本文を読み込み

**SKILL.md本文は選択後に読み込まれる**ため、本文に「いつ使うか」を書いても意味がない。

## 構成要素

良いdescriptionは3つの要素を含む:

### 1. 機能説明（何をするか）
```
PDFファイルからテキスト・表を抽出し、フォーム入力・文書マージを行う。
```

### 2. トリガーフレーズ（いつ使うか）
```
「PDFからテキスト抽出」「PDFフォームに入力」「PDF結合」と言われた時に使用。
```

### 3. 対象ファイル/ドメイン（該当する場合）
```
.pdfファイル、PDFフォーム、PDFドキュメントを扱う場合。
```

## 良い例・悪い例

### ドキュメント処理系

**良い例:**
```yaml
description: Word文書の作成・編集・分析。追跡変更、コメント、書式保持、テキスト抽出をサポート。「Wordファイルを作成」「.docxを編集」「文書にコメント追加」「追跡変更を有効化」と言われた時に使用。
```

**悪い例:**
```yaml
description: ドキュメントを処理します  # 何のドキュメント? どう処理?
description: Word文書を扱うのに便利です  # トリガーフレーズがない
```

### 開発ツール系

**良い例:**
```yaml
description: gitの差分を分析して説明的なコミットメッセージを生成。「コミットメッセージを作成」「変更をコミット」「コミットを書いて」と言われた時に使用。
```

**悪い例:**
```yaml
description: コミットをサポートします  # 何をどうサポート?
```

### データ分析系

**良い例:**
```yaml
description: BigQueryでデータ分析を実行。売上、財務、ユーザー行動のメトリクス取得。「今月の売上は?」「DAUを教えて」「収益レポート作成」などのデータクエリに使用。
```

**悪い例:**
```yaml
description: データを分析します  # どのデータ? どう分析?
```

## ドメイン別テンプレート

### ファイル処理スキル
```yaml
description: [ファイル形式]の[操作一覧]を行う。「[操作1]」「[操作2]」「[操作3]」と言われた時、または[ファイル拡張子]ファイルを扱う場合に使用。
```

### ワークフロー自動化スキル
```yaml
description: [ワークフロー名]を自動化。[主要機能]をサポート。「[トリガー1]」「[トリガー2]」と言われた時に使用。
```

### ドメイン知識スキル
```yaml
description: [ドメイン]に関する[知識タイプ]を提供。[対象データ/システム]について質問された時に使用。例:「[質問例1]」「[質問例2]」
```

## 制約

- 最大1024文字
- XMLタグ禁止
- 予約語禁止（「anthropic」「claude」）
- 三人称で記述
